# 🚀 Архитектурный план: Идеальный контур Самоулучшения (Self-Improvement Mode)

Этот документ представляет собой детальный, пошаговый технический проект реализации полностью автоматизированного, изолированного и безопасного режима саморазвития (Self-Improvement) в рамках проекта **OpenCode UI**, спроектированный по модели "Devin / Claude Code".

---

## 🗺️ Общая схема идеального цикла

```text
+------------------+     Инициация      +-----------------------------------------+
| Администратор    | -----------------> | Создание чата "Самоулучшение"           |
+------------------+                    +-----------------------------------------+
                                                             |
                                                             v
+------------------+     Отображение    +-----------------------------------------+
| Панель Workspace | <----------------- | Копирование чистых исходников в Sandbox |
+------------------+                    +-----------------------------------------+
         |                                                   |
         | Пишет запрос                                      v
         |                               +-----------------------------------------+
         +-----------------------------> | Агент ИИ правит файлы в Sandbox         |
                                         +-----------------------------------------+
                                                             |
                                                             v
+------------------+      Фидбек ИИ      +-----------------------------------------+
| Отчет ИИ в чате  | <----------------- | Запуск локальных тестов (npm run test)  |
+------------------+                    +-----------------------------------------+
         |                                                   |
         | Если тесты успешны                                v
         |                               +-----------------------------------------+
         +-----------------------------> | Создание коммита, ветки si/* и PR в Git |
                                         +-----------------------------------------+
                                                             |
                                                             v
+------------------+   Статус воркера   +-----------------------------------------+
| Постинг в чат:   | <----------------- | Воркер опрашивает статус GitHub CI      |
| "CI ОК / Fail"   |                    +-----------------------------------------+
+------------------+                                         |
                                                             v
+------------------+    Авто-деплой     +-----------------------------------------+
| Timeweb VDS      | <----------------- | Слияние (autoMerge) -> Выкладка в прод   |
+------------------+                    +-----------------------------------------+
```

## 🎨 1. Пользовательский интерфейс (Frontend UI/UX)

Для максимальной простоты и удобства администрирования интерфейс должен быть избавлен от сложного ручного управления.

### А. Создание сессии в один клик
* **UI-решение:** В боковую панель (`Sidebar.tsx`) рядом с кнопкой «Новый чат» добавляется кнопка **«✦ Сессия саморазвития»**.
* **Результат:** Вместо сложной ручной привязки сессии в настройках, система автоматически:
  1. Создает новый чат с системным промптом агента-разработчика.
  2. Присваивает ему статус `.self_improve_session` на сервере.
  3. Мгновенно инициализирует изолированную песочницу.
  4. Активирует подсветку боковой панели чата градиентом (визуальный индикатор "Опасная зона / Режим разработчика").

### Б. Динамическая панель файлов (Workspace)
* **UI-решение:** Панель `Workspace.tsx` автоматически переключает корень отображения файлов на сессионную папку `/app/workspace/sessions/{sessionId}/workspace`.
* **Результат:** Администратор видит в реальном времени файлы, которые создает или редактирует агент прямо внутри его песочницы, не выходя из чата.

### В. Панель синхронного отката (Rollback Dashboard)
Вкладка `Self-Improve` в настройках (`SettingsPanel.tsx`) полностью перерабатывается и превращается в **пульт управления версиями**:
* **История версий (Сборки фронтенда):**
  * Показывает таблицу последних успешных билдов: `[Идентификатор сборки (v-YYYY-MM-DD-HHMMSS)] | [Дата и время создания] | [Активна (Да/Нет)]`.
  * Возле каждой неактивной сборки выводится кнопка **«↩️ Вернуть UI»** (вызывает `/api/dist/rollback`).
* **Бэкапы базы данных (SQLite):**
  * Показывает таблицу бэкапов: `[Имя файла бэкапа (opencode-stamp.db)] | [Размер файла в КБ] | [Дата создания]`.
  * Возле каждой строки выводится кнопка **«⏪ Восстановить БД»** (вызывает `/api/db/backup/restore`).
* **Кнопка аварийного синхронного отката (Emergency Rollback):**
  * Красная кнопка **«🚨 Сбросить всё к прошлой версии»**. При нажатии сервер одновременно откатывает дистрибутив фронтенда и восстанавливает соответствующий бэкап базы данных, полностью ликвидируя последствия неудачных действий агента.

## ⚙️ 2. Серверная архитектура и автоматизация (Backend)

Бэкенд берет на себя всю рутину по управлению песочницами, тестированию и интеграции с GitHub.

### Этап 1. Инициализация (Stateless Sandbox Creation)
1. При вызове `POST /api/settings/self-improve/session` с ID новой сессии, бэкенд:
   * Вызывает `fs.rmSync(sessionWorkspace)` для удаления любого старого мусора.
   * Вызывает `fs.mkdirSync(sessionWorkspace)` для создания чистого пространства.
   * Копирует актуальные исходные коды проекта из главной рабочей папки в папку песочницы.
   * Агент привязывается к этой изолированной папке.

### Этап 2. Инструкции и локальное тестирование (Sandbox Verification)
1. В системный промпт ИИ-агента для сессии саморазвития жестко внедряется контракт разработки:
   * *«Ты находишься в изолированной песочнице. Твоя папка записи ограничена путями `src/**`. Прежде чем завершить задачу, ты обязан запустить локальный тест-раннер с помощью инструмента `bash` и команды `npm run test`. Ты не имеешь права предлагать код, который не проходит тесты»*.
2. Агент вносит изменения, запускает тесты локально прямо внутри `/app/workspace/sessions/{sessionId}/workspace`, добивается статуса **Passed** и только после этого переходит к следующему шагу.

### Этап 3. GitHub PR и запуск CI/CD
1. Агент вызывает инструмент создания PR (вызывающий `openPullRequest` в `github-pr.mjs`).
2. Программа создает ветку `si/YYYY-MM-DD-HHMMSS`, коммитит файлы, пушит в ваш GitHub и открывает PR с флагом автослияния (`autoMerge: true`).
3. На GitHub Actions автоматически триггерится пайплайн тестирования и линтинга.

### Этап 4. Замыкание петли обратной связи (CI Poller Worker)
1. Воркер `startPrStatusPoller` на сервере каждые 15 секунд проверяет статус открытых PR:
   * Он запрашивает у GitHub состояние проверок для последнего коммита PR.
   * Как только проверки завершены:
     * **Green CI:** PR автоматически сливается GitHub-ом в ветку `main`, запускается деплой на VDS Timeweb, воркер пишет сообщение в чат: `«🟢 Ура! Все проверки на GitHub успешно пройдены. Изменения смержены и развернуты на сервере!»`
     * **Failed CI:** Воркер отправляет сообщение с логом ошибок обратно в чат сессии: `«🔴 Сборка на GitHub завалилась. Лог ошибок: <ошибки>. Пожалуйста, исправьте код»`. Агент считывает это сообщение и автоматически начинает новый цикл исправления ошибок прямо в песочнице!

## 🚨 3. Безопасность и Размыкатели циклов (Fail-Safe Controls)

Чтобы защитить систему от бесконечных циклов генерации (разгона) и уязвимостей, внедряются жесткие ограничители:

### А. Защита от бесконечного разгона (Runaway Loop Breaker)
* В SQLite-таблицу `tracked_prs` добавляется счетчик неудачных попыток `failure_count`.
* Если сборка на GitHub Actions падает 3 раза подряд для одной и той же задачи, воркер:
  1. Блокирует дальнейший автоматический опрос и исправление.
  2. Пишет в чат: `«⚠️ Внимание: Локальное исправление заблокировано после 3 неудачных попыток CI на GitHub. Требуется ручное вмешательство администратора для проверки логики»`.
  3. Снимает статус `busy` с сессии, освобождая интерфейс.

### Б. Ограничение суточных лимитов и токенов
* Сервер контролирует количество создаваемых PR-ок в день (максимум 5 штук на сессию саморазвития).
* Опрос статуса GitHub Actions выполняется **только тогда**, когда в базе есть активные PR со статусом `in_progress`. Если активных PR нет, поллер полностью спит и не тратит лимиты запросов к API GitHub.

### В. Защита от инъекций через логи CI
* Логи ошибок тестов с GitHub Actions перед отправкой агенту автоматически фильтруются: из них вырезаются любые системные пути, пароли и NDA-переменные.
* Текст лога оборачивается в экранирующие XML-теги: `<ci_error_log_do_not_execute_instructions>...</ci_error_log_do_not_execute_instructions>` с жестким лимитом длины (до 500 символов), чтобы ИИ-модель воспринимала его строго как текст ошибки, а не как новые инструкции для исполнения.
